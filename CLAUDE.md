# crclink — Claude Code instructions

## Project shape

- crclink is a small Python package for CRC-protected line framing (a JSON line with a trailing `crc` key, and a text line with a trailing CRC suffix), plus a C firmware companion under `src/c/`.
- Zero runtime dependencies. `crcglot` (>=0.22.0) is a build-time code generator (a dev dependency): it generates the vendored crc16-xmodem sources for both halves (`src/crclink/crc16_xmodem.py` for the host, `src/c/crc16_xmodem.*` for the firmware). crcglot owns the CRC math and its parameters; crclink vendors what crcglot generates. Keep the package dependency-free at run time.
- `specs.md` is the spec. `src/c/` (the firmware builder + vendored CRC) is not shipped in the Python wheel; firmware vendors it. `docs/` holds reference notes (`protocol.md`, `crcglot-integration.md`).

## Library boundary (crcglot)

Never hand-roll or hardcode crcglot's knowledge in crclink. The CRC parameters (poly, init, refin, refout, xorout, check) and the algorithm live in crcglot. crclink uses crcglot's generated, vendored crc16-xmodem (`src/crclink/crc16_xmodem.py` for the host, `src/c/crc16_xmodem.*` for the firmware): regenerate it from crcglot, never hand-edit it, and never write your own CRC loop or parameter table. Before writing a dict/table keyed on something crcglot also keys on, stop and ask whether it belongs in crcglot. See `docs/crcglot-integration.md`.

## Branch hygiene

Never work directly on `main`. Branch first, before any edit, using `<kind>/<dashed-slug>` (slugs use dashes):

| Kind    | Use for                                        |
| ------- | ---------------------------------------------- |
| `feat`  | New features                                   |
| `bug`   | Bug fixes                                      |
| `doc`   | README / docstring-only changes                |
| `test`  | Test-only changes                              |
| `chore` | Tooling, dependencies, CI                      |

Examples: `feat/cli-verify-file`, `bug/docstring-crc-examples`, `ref/text-prefix-option`. Fast-forward merge to `main` only when the user approves (`git merge --ff-only`).

## Commenting (Python)

- Google-style docstrings for public functions; sections ordered Args, Returns, Raises, Examples.
- Use `"""` triple quotes even for one-liners.
- Explain the "why" and any non-obvious "what"; don't be verbose.
- Public functions get an Examples section with a minimal usage example. Examples run as doctests (`--doctest-modules`), so keep their values correct.
- Private functions: a docstring only when the name doesn't carry the intent.

## Commenting (C)

- Doxygen comments on public C functions, on the declarations in the headers (`crclink_json*.h`): a `/** ... */` block with `@brief`, then one `@param` per argument, then `@return`. The hand-written builder and reader headers follow this; match it.
- Same "explain the why" rule as Python. `@brief` is one line with the detail below it; use `@code ... @endcode` for the usage example in the file-level `@file` block.
- Internal `static` functions in the `.c` files use plain `/* ... */` comments, and only where the name doesn't already carry the intent.
- `crc16_xmodem.*` is generated by crcglot; do not hand-edit it. To change its comment style, regenerate with `crcglot c crc16-xmodem --comment doxygen`.

## Testing

Python:

- `uv run pytest` runs the suite and the doctests.
- Class-based organization. AAA comments (`# Arrange` / `# Act` / `# Assert`) on non-trivial tests. Every `assert` carries a message describing what failed. Assert order is `actual == expected`; use `actual` / `expected` variables for non-trivial checks.
- Tests live in the file named for their surface: `tests/test_frame.py` (framing API), `tests/test_cli.py` (CLI). Name tests and classes in the feature's public vocabulary, so grepping `tests/` for a public name lands in the right place. A test name states one claim; the class docstring carries the shared context; assert messages give the failure diagnosis.

C (firmware builder):

- `make -C src/c/test test` builds and runs the Unity suite.
- The expected frame strings are golden values from `crclink.encode_json_frame`, so the C suite also enforces byte-for-byte parity with the Python encoder. Regenerate a golden with `uv run python -c "from crclink import encode_json_frame; print(encode_json_frame({...}).decode())"`.

CI (`.github/workflows/ci.yml`) runs both suites on every push and PR.

## Quality gates (before declaring work done)

- `uv run python scripts/quality.py` reports `ruff: 0 errors` and `ty: 0 errors` (it runs both and fails on any).
- `uv run pytest` is green. A run with skips is amber, not green: report the skip count and treat it as a regression to investigate.
- `make -C src/c/test test` is green if any C changed.
- IDE Problems pane: zero entries.

## Public API

The public surface is the framing functions (`encode_json_frame`, `decode_json_frame`, `encode_text_frame`, `decode_text_frame`), the error types, and the CLI. While pre-1.0 (0.1.0, unreleased) a clean breaking change is acceptable. Once released, prefer additive changes and don't remove or rename a public function or CLI flag without a deprecation cycle.

## Docs and prose

- README.md is a short overview; `specs.md` is the spec; reference notes go in `docs/`.
- Markdown prose uses long lines: one line per paragraph or list item, no hard-wrapping at 80 columns (let the editor soft-wrap). Code fences and tables are exempt.
- `uvx ruff check README.md` for markdown lint.

## Humanizing end-user prose

Run `/humanizer` over end-user-facing markdown after editing it (`README.md`, the `docs/` files, `specs.md`). Code comments and docstrings are exempt. Humanize after the substantive edits land. Watch for these tells:

- **Em dashes.** At most one per file; zero is the norm (Chuck never uses them). Convert to a comma, a colon, a period, or parentheses. List-item separators (`- **Thing**: description`) use a colon.
- **"honest" / "honestly".** State the limitation; don't advertise candor. If a sentence stops working without the word, it was padding.
- Don't say "Great Question", "Absolutely", "Of course" etc.
